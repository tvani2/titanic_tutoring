### მონაცემთა გაწმენდა და დამუშავება (Cleaning & Preprocessing)

#### 1. გამოტოვებული მნიშვნელობების (NA) მქონე სვეტების დადროპვა
სვეტები, სადაც missing value-ების პროცენტი ძალიან მაღალი იყო, სრულად ამოვშალე:
* **Cabin (77% გამოტოვებული):** როდესაც სვეტის 70-75%-ზე მეტი მონაცემი აკლია, მისი შევსება რაიმე მნიშვნელობით ხშირად უფრო მეტ "ხმაურს" (noise) შემოიტანს მოდელში, ვიდრე სასარგებლო ინფორმაციას. ასეთ დროს მოდელი იწყებს არარსებულ კანონზომიერებებზე სწავლას, რაც იწვევს overfitting-ს.

#### 2. NA მნიშვნელობების შევსების სტრატეგია
ColumnTransformer-ის მეშვეობით გამოვიყენე შემდეგი მიდგომები:
* **რიცხვითი სვეტები (Age, Fare) შევავსე მედიანით:** საშუალო არითმეტიკულის ნაცვლად მედიანა ავირჩიე, რადგან ის უფრო მდგრადია outliers-ების მიმართ. მაგალითად, ასაკის შემთხვევაში, რამდენიმე ძალიან მოხუცი მგზავრი საშუალო მაჩვენებელს მაღლა სწევს, მედიანა კი რეალურ "შუა წერტილს" უკეთ ასახავს.
* **კატეგორიული სვეტები (Embarked) შევავსე მოდით (Mode):** ვინაიდან ეს ტექსტური მონაცემია, ყველაზე ლოგიკურია, გამოტოვებული ადგილი შევავსოთ იმ მნიშვნელობით, რომელიც ყველაზე ხშირად გვხვდება მონაცემთა ბაზაში.

#### 3. უსარგებლო სვეტების მოცილება
* **Name, Ticket, PassengerId:** ეს სვეტები მხოლოდ ინდივიდუალური იდენტიფიკატორებია. მათ არ გააჩნიათ სტატისტიკური კავშირი გადარჩენის ალბათობასთან (მაგალითად, ბილეთის ნომერი შემთხვევითი სიმბოლოების ერთობლიობაა). მათი დატოვება მოდელს დააბნევდა.

### Feature Engineering

* **FamilySize:** შევქმენი ახალი სვეტი ფორმულით: $FamilySize = SibSp + Parch + 1$. ეს გვეხმარება დავინახოთ მგზავრი მარტო იყო თუ ოჯახთან ერთად, რაც გადარჩენის შანსებზე პირდაპირ ახდენდა გავლენას.
 <img width="790" height="490" alt="image" src="https://github.com/user-attachments/assets/b8c97af9-988c-4a92-b1c7-398fa4654e89" />

* **WOE (Weight of Evidence):** გამოვიყენე **Pclass**-ზე. ეს მეთოდი კატეგორიულ ცვლადს გარდაქმნის რიცხვად იმის მიხედვით, თუ რამდენად ძლიერად არის ეს კატეგორია დაკავშირებული სამიზნე ცვლადთან (გადარჩენასთან).
  
  <img width="1189" height="490" alt="image" src="https://github.com/user-attachments/assets/e78cc7ed-921b-477e-9857-0c548612ac43" />

როგორც ვხედავთ, PClass = 1 მაღალი გადარჩენის ალბათობა აქვს, რის გამოც ეს feature გამოსადაგია. იგივე შეიძლება ითქვას სქესზეც - ქალების გადარჩენა 4-ჯერ უფრო მოსალოდნელია, ვიდრე კაცების.  

* **OneHotEncoder:** გამოვიყენე **Sex** და **Embarked** სვეტებზე, რათა ტექსტური კატეგორიები გარდაიქმნას 0 და 1 ფორმატში.

### Feature Selection

პროექტში გამოვიყენე სამეტაპიანი ფილტრაცია საუკეთესო ცვლადების დასატოვებლად:

#### 1. RFE (Recursive Feature Elimination)
Logistic regression-ის გამოყენებით შევარჩიე 8 ყველაზე მნიშვნელოვანი ცვლადი. RFE თანდათან აკლებს ყველაზე სუსტ ცვლადებს, სანამ სასურველ რაოდენობას არ მიაღწევს. 

#### 2. კორელაციის ფილტრი (Threshold = 0.85)
ამოვიღე ის ცვლადები, რომელთა შორის კორელაცია 0.85-ზე მეტი იყო. 
თუ ორი ცვლადი ერთმანეთთან ძალიან ძლიერადაა დაკავშირებული (მაგალითად Sex_male და Sex_female), ისინი მოდელისთვის ერთსა და იმავე ინფორმაციას ატარებენ (redundancy). ეს კი აფუჭებს მოდელის ინტერპრეტაციას. 

<img width="934" height="790" alt="image" src="https://github.com/user-attachments/assets/bf37b767-23fd-44c9-99b4-61e0e4949a67" />


#### 3. IV (Information Value) 
Information Value ზომავს ცვლადის "პროგნოზულ ძალას". Threshold-ად ავიყე 0.02.
სტანდარტულად, თუ $IV < 0.02$, ცვლადი ითვლება "არაპროგნოზულად".

| ცვლადი | IV ქულა | სიძლიერე |
| :--- | :--- | :--- |
| Sex_female | 1.3682 | ძალიან ძლიერი |
| Pclass | 0.5224 | ძლიერი |
| SibSp | 0.1866 | საშუალო |

### როგორ ვაკეთებდი Threshold-ის ოპტიმიზაციას?

იდეალური threshold-ის პოვნა არის ბალანსი ინფორმაციის შენარჩუნებასა და ხმაურის მოცილებას შორის.

* **თუ threshold-ს გავზრდით (მაგ. კორელაცია 0.95-მდე):** მოდელში დარჩება ძალიან ბევრი მსგავსი ცვლადი. შედეგად, მოდელი გახდება ზედმეტად რთული და მოხდება **Overfitting** (ტრენინგზე კარგი შედეგი, ტესტზე - ცუდი).
* **თუ threshold-ს შევამცირებთ (მაგ. კორელაცია 0.50-მდე):** ჩვენ წავშლით ბევრ ისეთ ცვლადს, რომლებსაც რეალურად სასარგებლო ინფორმაცია მოჰქონდათ. შედეგად მივიღებთ **Underfitting**-ს (მოდელი იქნება ზედმეტად მარტივი და ვერ ისწავლის კანონზომიერებებს).
ამ ინფორმაციის გამოყენებით, მოვსინჯე სხვადასხვა threshold და ვიპოვე საუკეთესო.

## ტრენინგი და ექსპერიმენტები

გამოვიყენე ორი ძირითადი ალგორითმი: logistic regression და decision trees.
ყველა ექსპერიმენტი დავარეგისტრირე MLflow-ში DagsHub-ზე. თითოეული მოდელისთვის დავლოგე შემდეგი მეტრიკები:

- `train_accuracy`, `val_accuracy`, `test_accuracy`
- `train_f1`, `val_f1`, `test_f1`
- `train_roc_auc`, `val_roc_auc`, `test_roc_auc`
- `train_precision`, `val_precision`, `test_precision`
- `train_recall`, `val_recall`, `test_recall`

### 🔹 Logistic Regression

შევამოწმე შემდეგი კომბინაციები:
- `C`: 0.01, 0.1, 1.0, 10.0
- `solver`: lbfgs, liblinear, saga
- `penalty`: l1, l2

## გამოცდილება

- **Preprocessing pipeline**-ის მოდელისგან გამოყოფა და ცალ-ცალკე MLflow-ში დარეგისტრირება inference-ს მარტივად გასაგებს ხდის
- **RFE → კორელაციის ფილტრი → IV** თანმიმდევრობა უფრო სუფთა feature selection-ს იძლევა ვიდრე ნებისმიერი ცალკეული მეთოდი
- ყოველთვის **train/test split უნდა მოხდეს EDA-მდე**
- **ტრენინგისა და inference-ის ცალკე notebook-ებში შენახვა** workflow-ს სუფთად ინახავს
- საუკეთესო მოდელის არჩევისას მხოლოდ **raw accuracy** არასაიმედო სიგნალია
